# Harness engineering for a self-evolving pharma research assistant

A harness — the operational scaffolding around an LLM that turns raw next-token prediction into a reliable agent — has become the dominant design unit of frontier AI systems in 2025–2026, displacing both hand-authored prompt chains and rigid workflow graphs. **For a pharma chemical and analytical development assistant operating over ELN, LIMS, project reports, Gantt data, and literature, the defensible architecture is a Claude Code–style harness as the outer agent, with LangGraph subgraphs serving as compiled, audited tools for the regulated slice of the workflow, a Voyager/AWM-style procedural-memory layer for self-evolution, and DSPy or Trace/TextGrad optimizers run offline to tune prompts and skills.** This mirrors what the self-evolving-agents literature (Gao et al. 2025; Fang et al. 2025) identifies as the near-term sweet spot, what production coding agents (Claude Code, Codex, SWE-agent, OpenHands) have converged on, and what GAMP 5 2nd ed. + FDA's 2024–2025 PCCP pattern can accommodate. The rest of this report develops each design decision with the depth needed to begin implementation.

## 1. Harness engineering — fundamentals and state of the art

### 1.1 What "harness engineering" actually means

The canonical formulation, from LangChain's Vivek Trivedy (2026), is **"Agent = Model + Harness"** — if you are not the model, you are the harness. The metaphor is literal: an LLM is a horse; the harness (reins, traces, yoke) is what turns raw power into useful work. The term was popularized in early 2026 by **Mitchell Hashimoto** (HashiCorp), who described repeatedly fixing agent mistakes by *"engineering the harness"* rather than the prompt. It was codified by **OpenAI's Codex team** in *"Harness engineering: leveraging Codex in an agent-first world"* (Feb 2026), by **Anthropic's** *"Effective harnesses for long-running agents"* and *"Scaling Managed Agents: Decoupling the brain from the harness"* (2025–2026), and formalized academically in *Meta-Harness* (Lee et al., arXiv 2603.28052) and the *Agent Harness Survey* (Preprints.org 202604.0428), which models a harness as a six-tuple **H = (Environment, Tools, Context, State, Loop, Verification)**.

Harness engineering nests above, not beside, its neighbors. **Prompt engineering** ⊂ **context engineering** (Karpathy/Lütke, June 2025: *"filling the context window with just the right information for the next step"*) ⊂ **harness engineering** (the full infrastructure around the model). **Agent scaffolding** (METR, SWE-bench literature) is near-synonymous. **Chain-of-thought** is an internal reasoning technique orthogonal to harness design. Crucially, harness engineering subsumes *everything an LLM can't do alone*: tool dispatch, memory, error handling, permissioning, sub-agent spawning, verification, and multi-surface serving.

### 1.2 The eleven components of a production harness

Synthesizing across Anthropic, OpenAI, LangChain, Raschka's *Components of a Coding Agent*, and Chawla's *Anatomy of an Agent Harness*, a production harness has roughly eleven layers: **(1)** an orchestration loop — the "dumb while-loop" where all intelligence lives in the model; **(2)** a tool registry with schemas, validation, sandboxing, and result formatting; **(3)** MCP (Model Context Protocol) as the open tool-integration standard; **(4)** context management (compaction, observation masking, just-in-time retrieval, subagent delegation); **(5)** multi-timescale memory (session, project-level `CLAUDE.md`/`AGENTS.md`, episodic reflections, three-tier hierarchies); **(6)** routing and orchestration logic (Anthropic's five workflow primitives — prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer); **(7)** evaluation/verification loops (rules-based, visual, LLM-as-judge — *"a 2–3× quality multiplier"* per Claude Code's Boris Cherny); **(8)** reflection mechanisms (Reflexion-style episodic self-critique); **(9)** guardrails and error handling with four error classes (transient → retry, LLM-recoverable → ToolMessage, user-fixable → interrupt, unexpected → bubble); **(10)** sandboxed execution (Docker, VMs, bwrap/seatbelt); **(11)** multi-surface serving (CLI, IDE, web, Slack). Avi Chawla's and Phil Schmid's framing — **model = CPU, harness = OS** — captures the systems-level analogy.

### 1.3 Why harnesses enable self-improvement and emergent behavior

A well-designed harness converts a stateless predictor into something that looks goal-directed. Five mechanisms do the work. **Dynamic tool use** means the LLM, not a hard-coded DAG, chooses the next action based on the latest observation. **Iterative refinement** — OpenAI's team writes linter messages *"specifically to teach the fix, so every failure becomes context for the next attempt."* **Meta-reasoning** (as in LATS) uses the LM simultaneously as actor, value function, and optimizer. **Self-critique** (Reflexion) stores verbal reflections in an episodic buffer, enabling "verbal reinforcement learning" without weight updates. And **emergent self-harness evolution**: Meta-Harness (Lee et al. 2026) exposes traces to a Claude-Code-based proposer that writes better harnesses, reaching SOTA on TerminalBench-2. The empirical punchline from LangChain's TerminalBench-2 work — **same model weights, different harness, moves an agent from outside the top 30 to rank 5** — and Pi Research's Grok Code Fast 1 jumping from 6.7% → 68.3% on SWE-bench just by changing edit-tool format, establishes that **harness quality now dominates model-rank differences at the frontier**.

### 1.4 Coding-agent exemplars

**Claude Code (Anthropic)** is the reference. Its harness uses agentic search via glob+grep (which empirically beat RAG on internal benchmarks), a sandboxed bash tool, file tools (Read/Write/Edit/MultiEdit), three subagent execution models (Fork, Teammate, Worktree), a three-tier `CLAUDE.md` hierarchy, Markdown **Skills** with YAML frontmatter and *progressive disclosure* (HumanLayer's framing), slash commands, PreToolUse/PostToolUse hooks, a 40+-capability permission system, and a canonical **Gather-Act-Verify** micro-loop. When its source leaked in March 2026 it was ~512,000 LOC — the scale of a serious harness.

**SWE-agent (Yang et al., NeurIPS 2024, arXiv 2405.15793)** contributed the **Agent-Computer Interface (ACI)** concept — *"LM agents are a new category of end users with their own needs and abilities, and would benefit from specially-built interfaces."* ACI ingredients include a fixed-window file viewer, edit-with-linting (syntactically broken edits are rejected with the Pyflakes message returned), summarized search, and explicit navigation. Mini-SWE-agent (~100 lines of Python) achieves 65% on SWE-bench Verified — proof of how much leverage correct ACI gives. **OpenAI Codex** has a three-layer harness (Codex Core + App Server + Client surfaces) with a strict prompt priority stack and cascading 32 KiB `AGENTS.md` files. **Devin (Cognition)** layers a planner, executor, shell, code editor, browser, and persistent workspace, with Agent Compute Unit budgeting. **Aider** is git-first, with a tree-sitter *repo map* ranked by PageRank over the dependency graph, diff-based edit formats, and two-model "architect mode." **OpenHands (ICLR 2025, arXiv 2407.16741)** formalizes the event-stream abstraction (Action/Observation append-only log) that is now the canonical harness primitive. **Cursor** embeds the harness in the IDE itself.

### 1.5 Sensors, gates, and the Ashby framing

Birgitta Böckeler's Thoughtworks formalization (Fowler, April 2026) is the clearest conceptual frame. A harness regulates the agent through two complementary flows: **feedforward "guides"** steer before the agent acts (AGENTS.md, skills, LSP definitions, codemod recipes), and **feedback "sensors"** observe after (tests, linters, type-checkers, screenshots, code-coverage, log-anomaly detectors, LLM-as-judge reviews). Each is either **computational** (deterministic, millisecond latency — ESLint, mypy, Pytest) or **inferential** (probabilistic — LLM judges, review subagents). **Gates** are the branching decisions these sensors feed: conditional routing, self-verification, permission gates (Claude Code's 40+ discrete capabilities), HITL interrupts, tripwires, and budget caps. Böckeler invokes **Ashby's Law of Requisite Variety**: *"a regulator must have at least as much variety as the system it governs."* Because LLMs can emit almost anything, the harness reduces variety by committing to a topology (a guide) and layering sensors on top — a variety-reduction move.

### 1.6 Feedback loops — the research backbone

The canonical lineage runs **CoT (Wei 2022) → Self-Consistency (Wang 2023) → ReAct (Yao ICLR 2023) → Self-Refine (Madaan 2023) → Reflexion (Shinn NeurIPS 2023) → Tree-of-Thoughts (Yao 2023) → LATS (Zhou ICML 2024) → Self-Discover (Zhou NeurIPS 2024)**, overlaid with **CoALA (Sumers et al. TMLR 2024, arXiv 2309.02427)** as the cognitive-architecture formalization. Reflexion's canonical claim — *"Reflexion agents verbally reflect on task feedback signals, then maintain their own reflective text in an episodic memory buffer"* — is the template for a pharma research assistant that must remember which analytical investigations succeeded and which dead-ends to avoid. At minimum, the pharma harness should implement a ReAct core plus Reflexion-style episodic memory; LATS-style tree search should stay in reserve for decisions with genuinely high branching factor, such as selecting an experimental design.

### 1.7 Harness vs. pipeline

Anthropic's *Building Effective Agents* (Dec 2024) drew the line cleanly: *"Workflows are systems where LLMs and tools are orchestrated through predefined code paths. Agents are systems where LLMs dynamically direct their own processes and tool usage."* A pipeline has known control flow, predictable latency, and easy debugging — but can only do what was enumerated at design time. A harness has emergent control flow, variable latency, and harder observability — but handles unbounded step counts, unknown branchings, and model upgrades. Anthropic's own practical rule: *"find the simplest solution possible; only increase complexity when needed."* A scientific research assistant will need both — deterministic workflows for routine data fetching, full harness loops for exploratory hypothesis-testing.

## 2. Harness engineering vs. LangGraph — a technical comparison

### 2.1 What LangGraph is

LangGraph 1.x (late 2025–2026) is a typed-state graph runtime. Its four primitives are **State** (a TypedDict or Pydantic model with `Annotated` reducer fields that behave like monoids/CRDTs when parallel nodes update the same key), **Nodes** (functions `State → State`), **Edges** (static or conditional, with `add_conditional_edges(node, router_fn, mapping)`), and **Compilation** (`builder.compile(checkpointer=..., store=..., interrupt_before=...)` produces an immutable CompiledGraph). Beyond that: **Command objects** collapse node+edge into a single returned `Command(goto=..., update=...)`; **checkpointing** (InMemory, Sqlite, AsyncPostgres) enables time travel and fault tolerance; **interrupts** (`interrupt(value=...)` + `Command(resume=...)`) give native HITL; **Subgraphs** and the `Send` API support fan-out map-reduce with reducer-based joins. LangGraph Platform, LangGraph Studio, and LangSmith close the loop with managed durable execution, live visualization, and evaluation.

**LangGraph's genuine strengths** are deterministic routing (every transition is code — indispensable for 21 CFR Part 11 auditability), durable execution via Postgres checkpointing (hour-long workflows survive restarts), explicit conditional-edge failure enumeration, Mermaid/Studio visualization for compliance review, a mature LangChain ecosystem (~700 tool wrappers as of LangChain's 2025 State of Agent Engineering survey of ~1,340 respondents), and safe parallelism via typed reducers. **Its genuine weaknesses** are topology rigidity (the compiled graph cannot be mutated at runtime — adding a new node requires a redeploy), maintenance burden that grows with every workflow ("graph sprawl" after ~50 workflows), a substantial framework tax (state schemas, reducers, `Annotated` types, Command semantics), imperative-glue nodes that reduce "the graph" to thin scheduling over imperative code, poor fit for truly novel queries (the system can only route *between* pre-declared nodes), and distributed-debugging complexity when parallel nodes touch overlapping state.

### 2.2 What a harness is architecturally

In contrast, the Claude Code/Codex/SWE-agent canonical form is a **~100-LOC while-loop** with MCP tools, subagent spawning, a skill library, context compaction, and permission governance — no explicit graph. Control flow emerges from the LLM's tool-call choices. Dynamic tool discovery via MCP means new data sources plug in without agent redeploys. Self-modifying behavior comes from skill files the agent itself writes to `.claude/skills/` and loads on demand. Subagent spawning provides runtime parallelism without compiled graphs. The Manus team's terminology — *"the harness floats; the pillar sinks"* — captures why every model upgrade automatically improves a harness but leaves a graph untouched.

### 2.3 Self-evolution — what the literature actually shows

Two 2025 surveys consolidate a now-mature research area. **Gao et al., *A Survey of Self-Evolving Agents* (arXiv:2507.21046, TMLR 2026)** taxonomizes 250+ papers across what/when/how/where to evolve. **Fang et al., *A Comprehensive Survey of Self-Evolving AI Agents* (arXiv:2508.07407)** frames the feedback loop as four components (System Inputs, Agent System, Environment, Optimizers). Key primary works the pharma harness should draw from:

**Voyager (Wang et al., arXiv 2305.16291, TMLR 2024)** — the canonical self-evolving agent. Its ever-growing skill library of *executable code* is the direct model for a pharma agent that accumulates validated analysis recipes. Voyager's skills transfer zero-shot to new worlds; a pharma equivalent is a skill for "generate a stability trend plot" that works across compounds. **Reflexion (Shinn et al. NeurIPS 2023)** showed +22 points on AlfWorld, +20 on HotPotQA, +11 on HumanEval purely from in-context verbal self-critique. **LATS (Zhou et al. ICML 2024)** reports 94.4% pass@1 on HumanEval, showing that MCTS sits cleanly above a ReAct loop. **Self-Discover (Zhou Google DeepMind, NeurIPS 2024)** has the LLM SELECT/ADAPT/IMPLEMENT atomic reasoning modules into a task-specific structure — the agent generates its own mini-graph at runtime. **Agent Workflow Memory (AWM, Wang et al., arXiv 2409.07429, ICLR 2025)** induces reusable workflows from past trajectories; +24.6% relative on Mind2Web, +51.1% on WebArena. **LEGOMem (AAMAS 2026)** extends AWM to multi-agent settings. **ReasoningBank (Google, ICLR 2026)** distills both successes and failures into structured memory, paired with memory-aware test-time scaling. **ADAS — Automated Design of Agentic Systems (Hu/Lu/Clune, arXiv 2408.08435, ICLR 2025)** — a meta-agent programs better agents in code, with inventions that transfer across domains and models. **DSPy, TextGrad (Nature 2024), Trace/OptoPrime (NeurIPS 2024)** are orthogonal optimizers that can compile prompts (and in TextGrad's case, arbitrary workflow components) via natural-language gradients.

**The synthesis**: procedural memory and skill libraries are the production-ready mechanism of self-improvement without weight updates; meta-agents designing agents in code are a credible near-term frontier but not yet appropriate for direct GxP deployment; automatic optimizers can post-process either a harness or a graph. **None of these require LangGraph; several are structurally awkward inside it** (LangGraph's compiled immutability is directly at odds with ADAS-style runtime topology change).

### 2.4 Hybrid patterns — the real production answer

The binary framing is a false dichotomy. Four hybrid patterns are viable:

**Pattern A (recommended for pharma) — Harness outer loop, LangGraph subgraphs as deterministic tools.** The agent harness exposes compiled LangGraph workflows as MCP-callable tools. The harness handles open-ended reasoning; the graphs handle well-characterized, GxP-validated workflows — adverse-event triage, batch-QC summarization, assay-plan approval with `interrupt()` for scientist sign-off. The harness decides *when* to invoke them; once invoked, execution is deterministic and auditable.

**Pattern B** — LangGraph outer loop with a harness-style planner node. Compliance-friendly outer frame with flexible interior.

**Pattern C** — LangGraph Platform purely as a durable-execution runtime for a single "harness node." You get Postgres-backed persistence and LangSmith tracing without conceding decision logic.

**Pattern D** — DSPy/Trace compilation over either. Orthogonal to the orchestration choice.

### 2.5 The framework spectrum in 2026

From most structured to most dynamic: **LangGraph ▶ LlamaIndex Workflows ▶ Pydantic AI ▶ CrewAI ▶ AutoGen/AG2 ▶ OpenAI Agents SDK ▶ Anthropic Agent SDK / Smolagents**. DSPy and Trace/TextGrad are orthogonal compilers. For an exploratory pharma R&D agent whose question distribution is genuinely open — a medicinal chemist asking, mid-project, *"find all analogs of compound X our competitors have patented in the last 18 months and cross-check binding-pocket feasibility against PDB structures"* — a pure-LangGraph system would be permanently-incomplete (constantly shipping new subgraphs) or trivially reduce to a single giant ReAct node. **The top SWE-bench systems (Claude Code, Anthropic's internal SWE agent, Codex, SWE-agent at ~65–76%) are harnesses, not compiled graphs**, and Manus's four rebuilds plus MorphLLM's 23 → 45% scaffold-driven improvement reinforce that harness quality dominates.

### 2.6 Comparison summary

| Dimension | LangGraph (pure) | Harness (Claude Code style) | Hybrid (harness + LangGraph subgraphs) |
|---|---|---|---|
| Control locus | Engineer-authored compiled graph | LLM chooses tools in a loop | Harness invokes deterministic sub-workflows |
| Adding new workflow | Code change, test, redeploy | Add MCP tool or skill file | Tool for dynamic; subgraph for validated |
| Novel queries | Only if path exists | Strong — emergent planning | Strong with auditable fallbacks |
| Determinism / audit | High | Low-to-moderate | High where regulated, flexible elsewhere |
| GxP / Part 11 fit | Native | Needs extra tooling | Best of both |
| Recovery from failures | Checkpointer built-in | Custom | LangGraph layer handles it |
| HITL | `interrupt()` native | Permission system | Native outer frame |
| Self-evolution | Weak (compiled immutability) | Strong (skills, AWM, subagents) | Strong within harness scope |
| Benefits from model upgrades | Partial | Direct | Direct in harness layer |
| Observability | LangSmith native | Build-your-own (OTel) | LangSmith + custom harness spans |
| Best for | Regulated, repeatable pipelines | Open-ended scientific reasoning | **Pharma R&D precisely** |

## 3. Pharma/chemistry-specific application

### 3a. Gate design

**Gate vs. guardrail vs. conditional edge.** Conditional edges (LangGraph) are control-flow primitives — *where* the agent goes next. Guardrails (NVIDIA NeMo Guardrails via Colang state machines, Guardrails AI with Guardrails Hub validators, LLM Guard, Amazon Bedrock Guardrails, Azure AI Content Safety) are cross-cutting safety layers — PII, prompt injection, toxicity, jailbreak. **Gates are domain-specific validation checkpoints that encode GxP/ICH/phase-appropriateness criteria** — the agent cannot advance until a gate passes. A robust harness layers all three: edges for routing, guardrails for generic safety, gates for pharma correctness. OpenAI Agents SDK's three "tripwire" gate types (input, output, tool) map directly onto this.

**Data quality gates anchor on ICH Q2(R2) and ICH Q14.** Q2(R2), adopted 1 Nov 2023 and effective 14 June 2024 (with FDA final guidance March 2024), defines the validatable performance characteristics — specificity, accuracy (with its new combined accuracy+precision confidence-interval criteria that ~75% of surveyed analysts flagged as an implementation concern), precision (%RSD ≤ 2.0% for API assay, ≤ 5–10% for impurities near QL), linearity/range (typically r² ≥ 0.995), DL/QL, robustness (with technology-specific expectations in Q2(R2) Tables 3–11 for HPLC, dissolution, CE, NIR, NMR, ICP-MS, multivariate). Q14 introduces the **Analytical Target Profile (ATP)** and **Method Operable Design Region (MODR)**, enabling a QbD-aligned lifecycle per Q8/Q9/Q12. **ALCOA+** (Attributable, Legible, Contemporaneous, Original, Accurate + Complete, Consistent, Enduring, Available) must be enforced as schema-level metadata on every ingested or emitted record. ALCOA++ adds Traceable.

**Regulatory compliance gates span GMP (21 CFR 210/211), GLP (21 CFR 58), GAMP 5 2nd ed. (2022)** — whose new **Appendix D11** extends to AI/ML, complemented by the standalone **ISPE GAMP® Guide: Artificial Intelligence (July 2025, ~290 pp.)** — **21 CFR Part 11 and EU Annex 11**, and emerging AI-specific guidance: **EMA's Reflection Paper on AI in the medicinal product lifecycle (EMA/CHMP/CVMP/83833/2023, finalized 9 Sept 2024)** with its "high patient risk"/"high regulatory impact" framing, draft **EMA/PIC/S Annex 22** on AI in pharmaceutical manufacturing (consultation closed Oct 2025, finalization mid-2026), and the **FDA Draft Guidance** (6 Jan 2025) *Considerations for the Use of AI to Support Regulatory Decision-Making for Drug and Biological Products*, which establishes a seven-step risk-based credibility framework (question of interest → context of use → model risk → assessment plan → execution → documentation → adequacy). The FDA/EMA joint *10 Guiding Principles of Good AI Practice in Drug Development* (2025) add human-centricity, proportional validation, and lifecycle monitoring.

**Scientific validity gates** encode DoE completeness (factorial completeness, p < 0.05, non-significant lack-of-fit, sufficient center points), stability (ICH Q1A(R2)–Q1E: long-term 25 °C/60% RH, intermediate 30 °C/65%, accelerated 40 °C/75%; 12-month long-term and 6-month accelerated minima at NDA; Q1E statistical evaluation for shelf-life with 95% one-sided CI vs. spec, Tukey poolability), and process capability (Cpk ≥ 1.33, or ≥ 1.67 for critical CQAs). **Stage-gate milestone gates** — pre-IND (preliminary method qualification, impurity levels within Q3A/B qualification thresholds), end-of-Phase 2 (CQAs locked, stability scope, container-closure lockdown, pre-meeting package ≥ 50 days ahead), pre-NDA (full method validation, typically 3 PV batches planned, primary stability committed) — are the project-level analogs. **Cross-functional gates** cover QC sign-off, QA release (Form CoA per Q7A content requirements), change control, and deviation investigation closure.

**Technical implementation defaults to a hybrid pattern**: the LLM extracts structured values from free-text reports, deterministic Python checks them against thresholds. For example, the agent reads a forced-degradation report, extracts peak-purity angle, mass balance, and unidentified-impurity levels; Python asserts mass balance 95–105% and no unidentified impurity > 0.10% (ICH Q3B). This inherits LLM flexibility on heterogeneous formats with deterministic, audit-ready pass/fail. Pure rule-based gates (Pydantic schema validators, SQL/regex constraint checks) are preferred for ICH thresholds, ALCOA+ metadata, and Part 11 signature manifestation; pure LLM-as-judge is reserved for semantic faithfulness and completeness-against-template, with **judge alignment benchmarked ≥ 85% vs. expert labels before production** (Ragas documents baselines around 75.6% improving with prompt iteration; DeepEval's `GEval` reports ~94% correlation with human evaluation).

**Gate failure handling** follows a four-level ladder: transient (retry with exponential backoff, cap at 2 per Stripe's reference), LLM-recoverable (return as ToolMessage), gate-hard-fail (halt, emit a GxP-compliant ALCOA++ deviation event to TrackWise/Veeva QualityOne/MasterControl with pre-populated context, create an HITL task for scientist/QA), policy/safety tripwire (immediate abort, security audit event). **Every gate evaluation must be persisted in a WORM-capable audit log** with gate ID, version (SemVer), input hashes, evaluator output (or LLM judge's raw reasoning), decision, NTP/UTC timestamp, user/agent identity, and signature manifest.

### 3b. Tool design and registry

The tool inventory spans ~50+ categories across ELN (Benchling, IDBS E-WorkBook, LabArchives, Revvity Signals, Dotmatics, Labguru, SciNote, eLabFTW), LIMS (LabWare, STARLIMS, Thermo SampleManager, LabVantage, Benchling LIMS, Dotmatics Nexxis), document/report parsing (Unstructured.io, LlamaParse, Azure Document Intelligence, AWS Textract, hybrid pgvector/Weaviate/Qdrant/Elasticsearch search), project management (MS Project via Graph, Smartsheet, Planisware, Planview, Jira/Asana), literature (PubMed E-utilities, Europe PMC, Semantic Scholar S2ORC ~136M papers, OpenAlex, Scopus, Web of Science, Crossref, Unpaywall), patents (USPTO PatentsView, EPO OPS, WIPO PatentScope, Google Patents Public Datasets, SureChEMBL, Reaxys with Pistachio reactions, CAS SciFinder-n), chemical databases (PubChem PUG-REST/PUG-View with ~5 req/s rate limit, ChEMBL, DrugBank), cheminformatics (RDKit, ChemAxon JChem/Marvin, OpenBabel, CDK, DECIMER for structure OCR), statistics (SciPy/statsmodels/pingouin, R for Q1E stability statistics, JMP/Minitab via JSL/Connect), and delivery (Microsoft Graph, Slack, python-docx/docxtpl/pandoc/WeasyPrint/ReportLab, Power BI/Tableau/Spotfire).

**Dynamic tool discovery should be built on Anthropic's Model Context Protocol (MCP)**, announced November 2024, now donated to the Linux Foundation's Agentic AI Foundation (co-founded with Block and OpenAI; backed by Google, Microsoft, AWS, Cloudflare, Bloomberg). The Nov 25 2025 spec revision added asynchronous operations, statelessness, and server identity; ~97M+ monthly SDK downloads and thousands of servers exist. MCP's three primitives (tools — model-controlled; resources — app-controlled; prompts — user-controlled) fit pharma cleanly. **Each source system gets an MCP server**; an internal enterprise registry (complementing the community MCP Registry launched Sept 2025) lists discoverable servers with capability metadata. Claude's **Tool Search** and **Programmatic Tool Calling**, and Anthropic/Cloudflare's **Code execution with MCP** pattern (where the agent writes code that filesystem-discovers tools on demand — reducing token use from ~150,000 to ~2,000 in documented cases, a 98.7% savings), are essential when hundreds of tools are connected. **Jenova.ai documented that past ~30 tools LLMs degrade, and at ~107 they fail outright**, making the code-execution-with-MCP pattern not just an optimization but a necessity for a lab-head agent.

**Tool versioning and capability metadata** must carry a semantic version (MAJOR for breaking I/O or pharma-interpretation changes, PATCH for non-decisional bug fixes), a **capability card** analogous to a model card (intended use per GAMP 5, JSON I/O schemas, side-effect classification — read-only/write/irreversible, **GxP-validation status with GAMP 5 category**, regulatory context of use per the FDA 2025 framing, auth scopes, rate limits, cost, data residency, and a link to the latest eval artifact), a deprecation policy with sunset date, and change-control linkage (any version bump of a GxP-validated tool triggers a change-control record).

**Security** requires identity propagation (the lab head's OAuth token passes through to Benchling/STARLIMS/Graph so native ACLs enforce project visibility), per-project vector-store partitioning (pre-retrieval ACL filters, not post-hoc — post-hoc leaks via embeddings), HashiCorp Vault/AWS Secrets Manager/Azure Key Vault for credentials, OAuth 2.0 + PKCE for user-delegated and mTLS for server-to-server GxP integrations, prompt-injection defense at the tool boundary (MCP's untrusted-description principle + LLM Guard/Bedrock Guardrails on returned tool content), ALCOA+-compliant tool-invocation logging, and TLS 1.3 + AES-256 + HSM-backed signing for e-signatures. **April 2025 MCP security research documented over-broad tool permissions, tool-spoofing, and prompt-injection vectors** — mitigated by signed server identity (Nov 2025 spec), per-tool permission scoping à la Claude Code's ~40-capability model, and a trusted-server allow-list for anything GxP.

### 3c. Knowledge and context management

**Context-window realities as of April 2026** favor a long-context-plus-RAG hybrid rather than either alone. **Claude Opus 4.7/4.6 and Sonnet 4.6** offer 1 M tokens GA (no long-context surcharge, ~600 image/PDF pages), with server-side compaction and file-system-based memory; Opus 4.6 needle-in-haystack recall drops from ~91.9 → 78.3 across the 1 M window. **GPT-5/5.2** offers 128 K–400 K depending on tier. **Gemini 2.5 Pro / 3 Pro / 3.1 Ultra** natively multimodal, 1 M default with 2 M on 3.1 Ultra, currently the most robust long-context retrieval among public models. A 200–400 page CMC project report plus 2–3 analytical development reports plus a stability study summary fits in-context on Opus 4.6+ or Gemini 3; a full LIMS export or a decade of ELN does not. **Lost-in-the-middle (Liu et al., Stanford 2023)** still applies — critical items belong at context start/end plus a structured scratchpad.

**The hybrid RAG stack** routes queries into three lanes. **Structured RAG** uses text-to-SQL with a semantic layer (Vanna AI 2.0 with RAG over DDL/docs/examples and row-level-security hooks; Cube, dbt-semantic-layer, or Snowflake Cortex Analyst as the metric/dimension vocabulary). Snowflake's benchmarks put naive LLM text-to-SQL at ~51% on enterprise workloads; semantic-layer-augmented approaches add 20–30 points. Critically, a query-validation allow-list (no DML, read-only role, mandatory `WHERE project_id/compound_id`) and a mandatory "show me the SQL" UI for sensitive queries are non-negotiable.

**Unstructured RAG's single highest-leverage upgrade is Anthropic's Contextual Retrieval (Sept 2024)** — prepend each chunk with a 50–100 token LLM-generated document-situating context before embedding and BM25 indexing. Anthropic reports **35% fewer retrieval failures with contextual embeddings alone, 49% with contextual embeddings + contextual BM25, and 67% combined with reranking** (top-150 → top-20), at ~$1.02 per million corpus tokens with prompt caching. **Late chunking**, **ColBERTv2**, and **RAPTOR** (recursive abstractive summary trees) complement for long protocols. **Hybrid lexical + dense (BM25 + embeddings + reciprocal-rank fusion) is non-negotiable for chemistry** — compound codes ("MK-1234"), impurity designators ("RRT 0.87"), batch numbers are exact-match-sensitive and fail pure semantic search. Multimodal handling: parse tables via Unstructured.io/Azure Document Intelligence/Gemini 3 native PDF into Markdown before chunking; caption figures with a vision LLM and store both image and caption embeddings; convert chemical structures to SMILES/InChI and embed with ChemBERTa/MolFormer; store spectra as structured artefacts, ideally ADF.

**Graph-based retrieval** is the right answer for the multi-hop queries pharma lives on — *"which projects used related compound classes; who is the SME; what methods were validated; what deviations occurred?"* **Microsoft GraphRAG (2024)** reports ~86% accuracy vs ~32% baseline RAG on enterprise datasets; **LazyGraphRAG** cuts indexing cost. **LightRAG** enables incremental updates — essential for daily ELN ingest. **HippoRAG 2 (NeurIPS 2024/2025)** is neurobiologically inspired (KG + Personalized PageRank), 10–30× cheaper for associative queries. **LlamaIndex PropertyGraph Index, Neo4j GraphRAG, TigerGraph** provide the infrastructure; AstraZeneca's Biological Insights Graph is a published industrial reference. **Ontologies anchor the graph**: ChEBI, MeSH, CHEMINF, BAO, OBI, NCIT for concepts; **Allotrope Data Format (ADF) with Allotrope Foundation Ontologies (AFO)** — HDF5 + RDF — for analytical data; **Pistoia Alliance FAIR principles**; AnIML and UDM. The Pistoia Alliance Methods Database (MSD + GSK, *J. Pharm. Biomed. Analysis* 2025) demonstrated vendor-agnostic HPLC-UV method transfer via ADF. Ingesting analytical data as ADF lets the agent reason natively over sample→instrument→method→result→operator without brittle parsers.

**Memory** follows Sumers/Yao/Narasimhan/Griffiths's **CoALA (TMLR 2024, arXiv 2309.02427)** four-memory decomposition. **Working memory** is the always-in-context scratchpad (§ below). **Episodic** (per-user/per-project traces) — candidate backends: **Letta/MemGPT** (Packer 2023; OS-inspired core/recall/archival tiers, LLM-managed via tool calls — best fit for self-hosted GxP), **Zep/Graphiti** (bi-temporal knowledge graph tracking *how facts change over time* — ideal for stability trends and project-state), **LangMem** (integrated with LangGraph, supports all three non-working types including procedural-memory self-rewriting), **Mem0** for SaaS contexts. 2025–26 LoCoMo benchmarks place Zep and Letta near the top (~85%, ~83%). **Semantic memory** is the hybrid RAG stack itself, anchored on AFO/ChEBI identifiers. **Procedural memory** is the Voyager skill library (executable code with self-verification) augmented by **AWM (Wang et al., arXiv 2409.07429)** inducing reusable textual workflows from past trajectories.

**Compression** stacks hierarchical/rolling summarization at project/series/experiment levels, importance-based retention (tag `regulatory_commitment`, `open_deviation`, `GMP_critical` and exclude from compression), Anthropic server-side compaction on Opus 4.7/4.6, Sonnet 4.6, and MemGPT-style virtual context swapping between core/recall/archival. **Traceability is first-class**: every retrieval chunk carries `source_uri`, `doc_hash`, `page`, `section`, and `ADF/ELN entity ID`, propagated through summaries and into final outputs; the generation prompt enforces "no unattributed factual sentence." Every agent response is stored with its exact prompt, retrieved context, tool calls, model version, and temperature — ALCOA+-treated as a lab record.

**The scientific working memory for a lab head** — always in context, reconstructed at session start — should include active projects (phase, current/next milestone, Gantt slip status), team composition per project, recent critical results (last 30 days: OOS, OOT, new impurity, unexpected polymorph), open deviations/CAPAs with due dates, regulatory submission timelines, active analytical methods per compound (with Q14 ATP identifier and validation status), stability studies with upcoming ICH pull points, reference-standard status, training/calibration status, and learned user preferences (report style, chart templates, units).

### 3d. Sensor design

**Sensors push; tools pull.** A sensor is triggered by an environment change (not an LLM call), is controlled by an external scheduler/broker, and emits asynchronous events into the agent's inbox — 0..n per unit time. Sensors are what separate a chatbot-over-ELN from a Monday-morning email saying *"Batch 24-017 hits the ICH 6-month timepoint Thursday; HPLC method v3 is not yet validated; I've drafted a method-transfer reminder to Analytics."*

The **pharma sensor catalogue** spans time-based (deadline proximity, Gantt slip, ICH Q1A/Q1B pull-points at 25/30/40°C conditions, regulatory-commitment deadlines, reference-standard/column/reagent expiry), data-arrival (new LIMS rows, ELN entries, HPLC/LC-MS/NMR/DSC/XRPD/Karl-Fischer/particle-size data, batch releases, CoAs — implemented via **Debezium CDC on LIMS Oracle/SQL Server logs** because most legacy LIMS have no webhooks), anomaly detection (OOS against spec, OOT against rolling EWMA/Shewhart, **Westgard multirules** — 1₃ₛ/2₂ₛ/R₄ₛ/4₁ₛ/10ₓ/7T — for QC; Cpk step-change; impurity-profile change; Isolation Forest/Autoencoder as second-line for higher-dimensional anomalies), dependency (prerequisite experiment done, method transferred, reference standard qualified, spec locked, stability protocol approved), literature/regulatory (daily PubMed/ChemRxiv/bioRxiv keyword+embedding match, patents, FDA/EMA/ICH RSS + classifier, ACS/AAPS/PittCon abstracts), and compliance (overdue batch-record review, past-due deviation investigation, training/calibration expiry, document-review-cycle, open audit findings).

**The technical backbone** is event-driven — Kafka, Redpanda, Redis Streams, or AWS EventBridge — with a topic per sensor family. Modern cloud ELN (Benchling, Signals) use webhooks; legacy LIMS (LabWare, STARLIMS, LabVantage) use Debezium CDC or timestamp-watermark polling. Instruments integrate via the **Allotrope ADF pipeline (Tetra Data Platform, ZONTAL Space)** which normalizes vendor raw data into ADF and emits events on new objects. **Temporal** is the right orchestrator for durable long-running stability timers (months to years); Prefect/Dagster for ingest DAGs. Edge cases that matter in practice: rate-limiting per sensor and user (prevent Monday avalanche), content-hash deduplication, idempotency keys on downstream actions, a heartbeat per sensor with a meta-sensor that fires on silence (silent failures are the worst failure mode in regulated environments), all events in UTC with source-TZ metadata for ICH timepoint math, and durable replay for post-restart catch-up.

**Triage** uses `priority = urgency × impact × confidence × user_preference_weight`, with three decision types: **interrupt** (OOS, safety — hard block, push notification), **queue to daily briefing** (Gantt slip, new lit), **passive log** (low-confidence anomalies). Focus mode only admits P0/P1 interrupts.

### 3e. Self-evolution and adaptation

**The central tension**: self-evolution and validated systems are in inherent conflict. The design resolution is **an outer fixed harness with an inner evolving core** — the harness is validated; its content evolves under a PCCP-style envelope, directly analogous to ICH Q12 post-approval change management with Established Conditions and allowed-change reporting categories.

**Feedback learning** uses thumbs-up/down with free-text "why," correction logs (user edits to agent-drafted memos are gold-standard signal), **DPO on preferences** over draft outputs (cheaper and more stable than PPO-RLHF for small teams), and reward-model distillation over rubric-graded rollouts aligned to Q14 ATPs, deviation-investigation completeness, and report templates. **Non-weight-updating optimization** is safer under GxP: **DSPy (MIPROv2, GEPA)** for compile-time optimization of stable pipelines (OOS-investigation drafter, stability-report writer); **TextGrad (Stanford, *Nature* 2024)** for instance-level test-time hard cases via "automatic differentiation through text"; **Trace/OptoPrime (Microsoft, NeurIPS 2024)**, PromptBreeder, APE as alternates. Empirical studies (arXiv 2506.19773, 2507.03620) show DSPy is most robust for multi-stage systems; TextGrad best on hard single-problem refinement. **Skill/tool library growth** uses Voyager-style acquisition — the agent writes Python, self-verifies on held-out cases, stores in a versioned registry — with each skill carrying metadata: author (human/agent), date, validation test, approval status, GxP-impact tag. **CodeAct (Wang 2024)** — executable code actions vs. JSON tool calls — outperforms on complex analytical data transforms.

**Workflow pattern recognition** runs weekly HDBSCAN over prompt embeddings; clusters crossing N occurrences trigger templating proposals (*"You've asked for a stability trend plot for Compound X every Monday for 6 weeks. Shall I schedule it?"*). **AWM** induces parametrized workflows from successful trajectories; users can name and pin plans that become procedural-memory entries with validated code skills underneath. **Ontology/KG expansion** uses InChIKey-exact plus string-similarity fuzzy entity resolution against ChEBI + internal registry; ambiguous matches go to a review queue. New compounds create KG nodes with proposed ontology class and properties. Schema changes are pull requests to the ontology, reviewed by a data-steward role — preserving open-world assumption handling without uncontrolled drift. LightRAG-style incremental indexing prevents nightly full re-indexes.

**Meta-agent patterns** use a LangGraph supervisor or AutoGen GroupChat delegating to specialists — Retriever, SQL-Analyst, Regulatory-Checker, Writer — with the Regulatory-Checker as a gate on outputs. **ADAS (Hu/Lu/Clune, ICLR 2025)** — meta-agents programming new agents in code — is a credible near-term frontier but **presently inappropriate for direct GxP deployment**; run it in a **shadow lane** against historical queries, human review of diffs and metric deltas, promotion only after approval.

**Safety and auditability** anchor on the **FDA PCCP** for AI/ML (final guidance Dec 2024, reissued Aug 2025 — *Marketing Submission Recommendations for a Predetermined Change Control Plan for AI-Enabled Device Software Functions*) with its three mandatory sections (Description of Modifications, Modification Protocol, Impact Assessment) as the right *pattern* even though drafted for devices: enumerate allowable self-evolution operations up front, bind each to pre-defined test gates. The **FDA Jan 2025 draft** and **GAMP 5 2nd ed. Appendix D11 + ISPE GAMP AI Guide (July 2025)** define the complete ML lifecycle. **EU AI Act (Reg. 2024/1689)** staged application — prohibitions Feb 2025, GPAI Aug 2025, high-risk Aug 2026 (extended to Aug 2027 for MDR/IVDR-embedded AI) — combined with EFPIA's 2025 position that pharma R&D AI is generally not high-risk under the Act (R&D exemption, no Annex III trigger), means a CMC agent faces GPAI transparency obligations plus sectoral GxP, with high-risk re-classification possible if outputs influence labelling/patient safety. The Commission's Feb 2026 high-risk guidelines will clarify.

Concrete practices: **SemVer git tracking** of every prompt/tool/skill/ontology/memory schema; **canary releases** to non-GxP-impacting sessions first; **instantaneous rollback** on regression; a **regression test suite on critical gates** (OOS handling, spec comparisons, stability-pull arithmetic, citation integrity, Q14 ATP reasoning, deviation classification) run on every self-modification, with failure blocking promotion; **red-teaming** (hallucinated peaks, fabricated batch numbers, prompt-injection via malicious PDF, data-exfiltration via tool use); **HITL approval with QA sign-off for any GxP-critical behavior change**, logged in an electronic change-control record; **ALCOA+ applied to the agent itself** — prompts, skills, memory as records subject to attribution/legibility/retention.

**The heuristic for when a learned skill crosses the line**: usage frequency × GxP-impact score beyond a threshold mandates promotion from procedural memory to a versioned, separately-validated tool with full CSV/CSA documentation.

## 4. Architectural recommendation and implementation path

**The recommended architecture is a hybrid — Anthropic Agent SDK (Claude Agent SDK) or a ~300-LOC custom harness as the outer agent loop, with LangGraph subgraphs compiled-once and validated-once for regulated workflows.**

The outer harness layer provides MCP tool servers for each data source (Benchling, STARLIMS, PubMed, patents, internal vector index, docking/ADMET services), so adding a new source is an MCP-server deployment, not an agent redeploy. A Postgres-backed skill-library table stores Voyager/AWM-style procedural memory, indexed by embedding and retrieved into context at task start; successes and ReasoningBank-style failures both. Subagent spawning (Claude Code's `Task` pattern) handles literature-wide searches and parallel docking. Aggressive context compaction via periodic summarization into scratch docs prevents Manus-style "context anxiety." Permission policies are declaratively encoded: deny-by-default for any LIMS/ELN write, HITL approval for schedule-changing actions.

The inner LangGraph layer — compiled immutable — owns adverse-event triage (with formal IQ/OQ/PQ), batch-QC summarization, assay-plan approval (with `interrupt()` for scientist sign-off, checkpointed in Postgres), and any other enumerable GxP-critical workflow. These are exposed to the outer harness as tools; the harness decides when to invoke them, the inner graph executes deterministically.

**Observability** stacks LangSmith for the LangGraph subgraphs and OpenTelemetry + Langfuse for the outer harness, unified by a cross-layer trace ID. A **weekly offline self-evolution job** runs AWM/ReasoningBank extraction from successful sessions, DSPy MIPROv2 or Trace/OptoPrime optimization against a curated validation set of ~200 scientist-rated past queries, and — behind a human-review gate — ADAS-style proposals for new sub-skill modules that ship to production only after approval.

**Model defaults**: Claude Opus 4.7 (1 M context, compaction, task budgets) for orchestration and drafting; Gemini 3 Pro for multimodal and very-long-context analysis; a smaller Sonnet/Flash-class model for sensor triage and cheap extraction. **RAG defaults**: hybrid router with Contextual Retrieval + BM25 + reranker (unstructured), Vanna-style text-to-SQL with a semantic layer (structured), GraphRAG on "gold" corpora + LightRAG/HippoRAG 2 on daily ELN ingest (relational), grounded on ChEBI + AFO + Pistoia-standard method ontology, ingesting analytical data as ADF. **Memory defaults**: Letta or LangMem for self-hosted episodic + procedural, Zep-style temporal KG for changing facts, explicit CoALA working/episodic/semantic/procedural separation. **Sensor defaults**: Debezium CDC on LIMS/ELN, webhooks where available, Kafka/Redpanda backbone, Temporal for durable stability timers, Westgard rules + Isolation Forest as the starting detector set. **Governance defaults**: GAMP 5 2nd ed. + Appendix D11 + ISPE GAMP AI Guide as the validation framework; a PCCP-style document enumerating allowable self-evolution without new CSV; EU Annex 22 and FDA Jan 2025 draft as forward-looking constraints.

## 5. Open questions and practical challenges

Several open questions will require empirical resolution during implementation. **How much to load vs. retrieve**: with 1–2 M-token models, full-project loading is feasible but expensive and degrades recall after 60–80% fill — the optimal split is task-dependent. **Graph cost vs. value**: GraphRAG indexing is LLM-expensive; a tiered approach (GraphRAG on SOPs, regulatory, final reports; LightRAG on daily ELN) is likely optimal but unproven at scale. **When does a learned skill cross from agent state to validated software**: the usage × GxP-impact threshold is a design choice, not a regulatory constant. **EU AI Act reclassification risk**: the Feb 2026 Commission high-risk guidelines may redraw the line for CMC agents feeding regulatory filings — architecture must allow re-classification without re-architecture. **Model drift vs. process drift**: when an OOT fires, is it the chemistry, the method, the instrument, or the agent's detector? Robust root-cause separation requires independent monitoring of each layer — the agent should not be both the detector and the sole diagnostician. **Memory privacy**: per-user episodic memory is personal data under GDPR, must be deletable on request without corrupting audit trails. **Judge-alignment calibration**: LLM-as-judge baselines at ~75.6% alignment require iterative prompt engineering to reach production-grade ≥ 85% — budget for this upfront. **Harness-layer evals are harder than graph-layer evals**; simulation-based evaluation infrastructure (Maxim, Langfuse, custom) is a day-one investment, not a later addition.

## 6. Conclusion

The shift from prompt engineering to context engineering to harness engineering is not a rebranding — it is a recognition that the infrastructure around the model is now the dominant variable in agent quality. **Agent = Model + Harness** is literally true; the LangChain TerminalBench-2 finding (same weights, different harness, 30+ ranks of movement) and Pi Research's 6.7 → 68.3% SWE-bench leap from an edit-tool format change make this empirically unambiguous. For a pharma CMC/analytical development assistant, the implication is that investing in harness components — MCP-based tool registry, hybrid RAG with contextual retrieval and graph layers, CoALA-structured memory, event-driven sensor backbone, hybrid gates with ALCOA+ audit trails, a Voyager/AWM skill library under a PCCP-style self-evolution envelope — will compound far more value than chasing the next model release or the next orchestration framework. The harness floats on the model; the pillar sinks. The system you build around the model is the system you keep.

Pure LangGraph over-fits to today's known workflows and under-serves the open-ended scientific question distribution a pharma research assistant must field. Pure harness under-serves regulated workflows where determinism and auditability are non-negotiable. **The hybrid — harness outer loop, LangGraph inner subgraphs for validated paths, skill library for accumulated expertise, MCP for extensible tools, PCCP-style governance for controlled evolution — is the defensible choice for 2026.** It preserves GxP compliance where regulation demands it, handles genuinely novel queries where science demands it, accumulates institutional knowledge where organizational learning demands it, and leaves a clean path toward more aggressive self-evolution (ADAS-style meta-design, co-evolutionary optimization) as both the research frontier and the regulatory guidance mature.
