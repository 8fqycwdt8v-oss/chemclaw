# Workflow Engine + Queue + Newest-Code Audit — 2026-05-03

## Executive Summary

The Phase 6–9 additions introduce three interlocking primitives: a Postgres-backed task queue (`services/queue/`), a standalone workflow engine (`services/workflow_engine/`), and an in-process recipe engine embedded inside `mcp-xtb` (`services/mcp_tools/mcp_xtb/workflow.py`). Together they represent a genuine architectural expansion that is broadly correct in its foundations but carries several merge-risk issues that deserve attention before these paths see production load.

The most significant structural tension is a **naming collision and conceptual divergence** between two things both called "workflow engine": the in-process `mcp_xtb/workflow.py` (synchronous request/response, subprocess-driving, no DB) and the Postgres-event-driven `services/workflow_engine/main.py` (asynchronous, event-sourced, multi-service fan-out). They have no direct integration path. The `conformer_ensemble` route that was changed in commit `c72dd92` routes through the in-process engine only — the Postgres-backed engine plays no role there. This divergence is not documented in either module.

The queue worker's correctness posture is sound: it uses `SELECT ... FOR UPDATE SKIP LOCKED`, has lease expiry reclaim, increments `attempts`, and gates retry on `max_attempts`. JWT minting is done correctly with per-service `aud` claims. The primary gap is dead-letter alerting and lack of any unit tests for the worker logic itself.

The `29_workflows.sql` schema is well-designed (event-sourced, bi-temporal definitions, separation of `workflow_events` canonical log from `workflow_state` materialized projection) but has a RLS gap: FORCE RLS is not enabled on any of the five workflow tables. Both `task_queue` and `chemspace_screens` have the same gap. The `27_job_queue.sql` comment promises "RLS / grants" but delivers only grants — no policies, no FORCE RLS, no row-level isolation. This is by design (chemistry data is tenant-agnostic), but the absence of FORCE RLS means a future engineer applying project-scope RLS to these tables will find it silently bypassed by `chemclaw` (the table owner).

The xtb workflow routing change (`c72dd92`) is sound for the in-process case: backward-compat `/conformer_ensemble` route is preserved, timeouts are capped, and inputs are validated by the `Inputs` Pydantic schema. The Boltzmann re-weighting now uses the correct RT(298 K) = 0.5925 kcal/mol and is well-tested. The one backward-compatibility break is semantic: the legacy route returned weights from CREST pre-opt energies; the new route returns post-opt energies. No flag or version indicator surfaces this change to callers.

Test coverage for the in-process engine is good (17 engine-semantics tests, 8 recipe tests). Test coverage for the Postgres-backed workflow engine is minimal (4 unit tests, pure helper coverage only). The queue worker has zero test coverage.

## As-Built Workflow Model

There are two distinct systems both referred to as "the workflow engine" and they must be understood separately.

**The in-process recipe engine** (`services/mcp_tools/mcp_xtb/workflow.py`) is a synchronous (request-scoped) engine that runs multi-step subprocess pipelines entirely within an HTTP request. A `Workflow` is a tuple of typed `Step` objects each carrying an async callable, an optional per-step timeout, and an `optional` flag. The engine runs steps sequentially (or fans out via `parallel_map` for the conformer optimisation step), threads a mutable `Ctx` object carrying `workdir`, `inputs`, and `artifacts` between steps, enforces per-step and total deadlines with `asyncio.wait_for`, and returns a `WorkflowResult` Pydantic model. There is no database involvement, no event log, and no persistence. The two registered recipes (`optimize_ensemble`, `reaction_energy`) demonstrate the pattern well. This engine lives entirely inside the `mcp-xtb` container.

**The Postgres-backed workflow engine** (`services/workflow_engine/main.py`) is a separate long-running service. A workflow is a JSON definition stored in the `workflows` table with bi-temporal versioning. Execution creates a `workflow_runs` row and drives it by appending `workflow_events` rows (the canonical log) and projecting state into `workflow_state` (the materialized cursor). The engine subscribes to `pg_notify('workflow_event')`, wakes on each new event plus a 30-second poll fallback, then sweeps all `running` workflow runs and advances each by executing its `next_index` step. Step kinds supported today in production are `tool_call` (HTTP POST to an MCP service) and `wait` (polling a `task_batches` row until completion). `conditional`, `loop`, `parallel`, and `sub_agent` steps are accepted by the Zod validator on the agent side but execute as a no-op in the Python engine, returning `{"note": "kind=... not yet executed; returning empty result"}` — this is a silent no-op, not a failure. The engine uses `chemclaw_service` credentials (BYPASSRLS) and mints per-service JWTs for the MCP calls it dispatches. There is no docker-compose entry for this service.

The agent-side interface consists of six TypeScript builtins (`workflow_define`, `workflow_run`, `workflow_inspect`, `workflow_pause_resume`, `workflow_modify`, `workflow_replay`) plus `promote_workflow_to_tool` (Phase 9) and `conformer_aware_kg_query` (Phase 7). All six operations are audited via `appendAudit`. The `startRun` path in `client.ts` correctly inserts the run as `status='running'` and fires a `start` event which the Python engine picks up.

## As-Built Queue Model

`services/queue/worker.py` is a single-process at-least-once task queue backed by the `task_queue` Postgres table. It uses `SELECT ... FOR UPDATE SKIP LOCKED` inside a CTE to atomically claim exactly one task per `_lease_one` call. The worker operates on a configurable list of `task_kind` prefixes (default: `qm,genchem,classifier`), runs a NOTIFY/poll loop, and limits concurrency via `asyncio.Semaphore(QUEUE_CONCURRENCY)`. Lease expiry reclaim runs at the top of every `_sweep_all` call, resetting leased-but-expired rows back to `pending`. Retry semantics: up to `max_attempts` retries (default 3); on the final attempt the row moves to `failed` with an `error` JSONB column. There is no dead-letter queue, no alerting on transition to `failed`, and no maximum backoff between retries (retries fire on the next `_sweep_all` immediately after lease reclaim). The `idempotency_key` unique constraint on `(task_kind, idempotency_key)` guards against duplicate enqueues from callers that set the key. There is also no docker-compose entry for the queue worker.

## Concerns Specific to xtb Workflow Routing (post-c72dd92)

The `conformer_ensemble` route now calls `workflow.run(RECIPES["optimize_ensemble"], ...)` in-process (not the Postgres-backed engine). This is the correct architectural choice for a request/response endpoint: the recipe runs inside the HTTP request, cleans up its temp directory, and returns a structured `WorkflowResult`. The five-step pipeline (embed → crest → parse → opt → boltzmann) is well-structured and the test coverage for its semantics is the best in the entire new subsystem.

Three specific concerns deserve tracking:

1. **CREST binary dependency not in `readyz`.** The `/readyz` check is `_xtb_available()` (checks for the `xtb` binary). If `crest` is missing from the image, the `conformer_ensemble` route will report itself healthy until a request arrives and fails at the `_crest` step. The recipe's `crest` step is not marked `optional`, so a missing binary causes a 200-response with `success=False` rather than a 503 at service startup. Adding `crest` to the readyz check is a one-line fix.

2. **Parallel `xtb` opt opens N subdirectories under a single shared `TemporaryDirectory`.** The `_opt` step creates `ctx.workdir / f"conf_{idx}"` for each conformer, running up to 4 xtb processes concurrently. This is safe — each subdir is independent — but the total temp-dir lifetime is bounded by the outer `async with tempfile.TemporaryDirectory()` in `workflow.run`. If any `parallel_map` task raises, `TaskGroup` cancels the rest and the exception propagates to the engine, which records `step_failed`. The temp dir is then cleaned up by the context manager even on failure. This flow is correct.

3. **Energy parse failure in `_opt_one` raises `ValueError` which collapses the entire `parallel_map` via the `ExceptionGroup` unwrap.** If one of the N parallel xtb optimizations fails to produce a parseable energy, the first exception cancels all remaining xtb processes and the whole ensemble fails, not just that conformer. For a 20-conformer run this means a single bad geometry aborts the other 19. The existing test (`test_optimize_ensemble_crest_failure_reports_step`) only tests a CREST-level failure, not a per-conformer xtb failure. A more defensive approach would mark individual conformer failures as warnings (optional sub-step) and drop that conformer rather than failing the run.

The highest-priority items for pre-production review are: (1) the missing docker-compose entries for both new services, (2) the silent no-op for unimplemented workflow step kinds, (3) the missing FORCE RLS declarations, and (4) the multi-replica race condition in `_sweep`. The in-process recipe engine and its test suite are the strongest part of this addition.
